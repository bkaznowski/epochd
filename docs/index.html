epochd

Inject a fake wall-clock time into any running Linux process — without restarting it, modifying its binary, or stopping it for more than a millisecond. Supports both advancing mode (clock shifts by a fixed offset and keeps ticking) and freeze mode (clock is pinned to an exact instant). Purpose-built for testing time-sensitive behaviour in Kubernetes pods and Go programs.

⚡

Live updates in ~1 µsAfter the one-time injection, changing the time costs a single process_vm_writev call — no ptrace stop required.

🎯

Per-pod precisionTarget specific pods by label selector. Other pods on the same node see real time.

🧪

Test-first designAuto-cleanup on test failure, TTL-gated timeshifts, and a WithTimeT helper that self-skips when no cluster is available.

📦

Pure Go, no CGoCross-compiles from any host. The trampoline is pre-assembled bytes embedded in the binary.

What problem does it solve?

Many distributed systems have time-dependent behaviour that's hard to test: certificate expiry, token refresh, scheduled job execution, cache TTL, session timeouts. The typical approaches all have serious drawbacks:

ApproachDrawback
Mock time.Now() in GoOnly works inside the same binary. Useless for testing behaviour across multiple services or in a running container.
Set the system clockAffects every process on the machine. Breaks TLS, kube-scheduler, monitoring, and anything else that cares about real time.
libfaketime / LD_PRELOADRequires restarting the target process. Doesn't work with statically linked binaries (which includes most Go programs).
Wait for real time to passA test that waits for a 24-hour expiry to fire is not a practical test suite.
Intercept every clock_gettime syscall with ptraceStops the process on every call. Go's runtime calls clock_gettime tens of thousands of times per second. Completely unusable.

epochd solves this by patching the vDSO's clock_gettime entry with a JMP to a small trampoline page that adds a fixed offset to the real time. The patch is invisible to the process, survives fork, and can be updated at any time without stopping the target.

How it works

Modern Linux maps a shared library called the vDSO ("virtual dynamic shared object") into every process's address space. Glibc's clock_gettime doesn't issue a syscall — it calls a symbol in the vDSO which reads timekeeping data from a shared kernel memory page. This is why clock_gettime takes ~20 ns instead of ~200 ns.

epochd exploits this by overwriting the first 5 bytes of the vDSO's clock_gettime entry with a JMP rel32 instruction pointing to a small trampoline page. The trampoline:

  1. Issues a real syscall to get the actual current time.
  2. Reads an offset from a small state struct embedded in the same page.
  3. Adds the offset to the returned timespec.
  4. Returns. The calling code never sees the trampoline.
[vdso region] clock_gettime entry: E9 xx xx xx xx ← JMP rel32 (patched by epochd) │ ▼ [anon rwx page, allocated by the target itself via mmap] ├─ trampoline code (86 bytes) │ issue real syscall → get real time │ load offsetSec / offsetNsec from state struct │ add offset to timespec │ return 0 └─ state struct (32 bytes, at byte offset 86) offsetSec int64 ← in advancing mode: offset from real time offsetNsec int64 in frozen mode: absolute frozen timestamp enabledMask uint64 (1 = advancing, 3 = frozen) generation uint32 (bumped on each update)

Updating the fake time requires writing 32 bytes into the state struct using process_vm_writev. No ptrace stop. No signal. The target process continues running.

Two clock modes are controlled via enabledMask:

  • Advancing (enabledMask = 1) — the trampoline adds (offsetSec, offsetNsec) to the real time returned by the syscall. The clock ticks forward at the normal rate, shifted by the injected offset.
  • Frozen (enabledMask = 3) — the trampoline ignores the real syscall result and returns the absolute timestamp stored in the offset fields. The clock does not advance.

The injection sequence

The one-time setup (on first InjectAtTime or Start call) involves three brief ptrace stops:

  1. vDSO discovery — parse /proc/<pid>/maps for [vdso], resolve the clock_gettime symbol via ELF.
  2. Remote mmap — while the target is ptrace-stopped, make it call mmap on its own behalf. The new page is allocated as close as possible to the vDSO entry (within the ±2 GB reach of JMP rel32).
  3. Patch and detach — write the trampoline payload, patch the 5-byte vDSO entry, then detach. The target resumes immediately.

⚠️

Production cluster advisory

Do not deploy epochd in a production cluster. The agent runs as a privileged pod with hostPID: true and the ability to inject code into any process on the node. It should only run in dedicated test clusters.

Kubernetes cluster — using the Go SDK

Import the SDK into your e2e test package:

go get epochd/pkg/sdk

WithTimeT — the recommended test helper

WithTimeT creates a timeshift, runs your assertions, then automatically deletes the timeshift via t.Cleanup — even if the test panics. It automatically skips when EPOCHD_URL is not set, so the same test can be run against a local unit test runner and a full cluster.

func TestCertificateExpiry(t *testing.T) {
    client := sdk.NewClient(os.Getenv("EPOCHD_URL"))

    // Shift all pods matching app=my-service to one year in the future.
    sdk.WithTimeT(t, client, "default", "app=my-service",
        time.Now().Add(365*24*time.Hour),
        30*time.Minute, // TTL — timeshift is auto-deleted after 30 min even if the test crashes
        func(t *testing.T, ts *sdk.Timeshift) {
            // Assertions run here. Pods matching "app=my-service" see time as ~1 year ahead.
            resp, err := myClient.GetStatus()
            if err != nil {
                t.Fatalf("expected no error, got: %v", err)
            }
            if resp.CertExpired {
                t.Error("certificate should not be expired yet")
            }
        },
    )
}

Manual create / delete

For more control, use CreateTimeshift and DeleteTimeshift directly:

client := sdk.NewClient(os.Getenv("EPOCHD_URL"))
ctx := context.Background()

// Create an advancing timeshift — clock shifts forward 2 years and keeps ticking.
ts, err := client.CreateTimeshift(ctx, "default", "app=payment-service",
    time.Now().Add(2*365*24*time.Hour), time.Hour)
if err != nil {
    log.Fatalf("create: %v", err)
}
fmt.Printf("timeshift %s applied to: %v\n", ts.ID, ts.AppliedTo)

// Advance time further by one day.
ts, err = client.AdvanceTimeshift(ctx, ts.ID, 24*time.Hour)

// Move time to a specific absolute point.
ts, err = client.UpdateTimeshift(ctx, ts.ID, time.Now().Add(3*365*24*time.Hour))

// Clean up (resets all injected processes to real time).
_ = client.DeleteTimeshift(ctx, ts.ID)

Freeze mode

In freeze mode the clock is pinned to a specific instant. Every clock_gettime call in the target returns exactly that time, regardless of how much real time passes.

frozenAt := time.Date(2030, 1, 1, 0, 0, 0, 0, time.UTC)

// Create a frozen timeshift — clock never advances.
ts, err := client.CreateFrozenTimeshift(ctx, "default", "app=web",
    frozenAt, time.Hour)

// Switch an existing timeshift to frozen mode at a new instant.
ts, err = client.FreezeTimeshift(ctx, ts.ID,
    time.Date(2030, 6, 1, 0, 0, 0, 0, time.UTC))

// Advance a frozen clock by one day (clock stays frozen at the new instant).
ts, err = client.AdvanceTimeshift(ctx, ts.ID, 24*time.Hour)

// Switch back to advancing mode.
ts, err = client.UpdateTimeshift(ctx, ts.ID, time.Now().Add(365*24*time.Hour))

WithFrozenTime — frozen test helper

func TestCertificatePinnedAt(t *testing.T) {
    client := sdk.NewClient(os.Getenv("EPOCHD_URL"))
    frozenAt := time.Date(2030, 1, 1, 0, 0, 0, 0, time.UTC)

    sdk.WithFrozenTime(t, client, "default", "app=my-service",
        frozenAt, 30*time.Minute,
        func(t *testing.T, ts *sdk.Timeshift) {
            // Every call to time.Now() in target pods returns exactly frozenAt.
            // ts.Time == frozenAt (constant; does not grow).
        },
    )
}

Preview which pods will be affected (dry run)

Use Resolve to see which pods and containers match a selector before creating a timeshift:

pods, err := client.Resolve(ctx, "default", "app=payment-service")
for _, pod := range pods {
    fmt.Printf("pod %s: containers %v\n", pod.Name, pod.Containers)
}

Wait for injection to be confirmed

WaitForActive polls /timeshifts/{id}/status every 250 ms until every container in the timeshift reports a successful injection (Status non-nil, no error). It returns immediately on a permanent per-container error.

WithTime and WithTimeT call WaitForActive automatically before invoking your callback — you do not need to sleep or poll manually when using those helpers. Call WaitForActive directly only when using CreateTimeshift on its own:

ts, err := client.CreateTimeshift(ctx, ns, selector, target, ttl)
if err != nil { log.Fatal(err) }

// Block until every container is confirmed injected.
if err := client.WaitForActive(ctx, ts.ID); err != nil {
    log.Fatalf("injection failed: %v", err)
}
// Safe to make assertions now.

Inspect live injection state

TimeshiftStatus queries each agent for the current trampoline state — useful to confirm the injection landed and what time each container currently sees:

status, err := client.TimeshiftStatus(ctx, ts.ID)
for _, c := range status.Containers {
    if c.Error != "" {
        fmt.Printf("%s/%s: ERROR %s\n", c.Pod, c.Container, c.Error)
        continue
    }
    fmt.Printf("%s/%s: lastTarget=%s gen=%d\n",
        c.Pod, c.Container, c.Status.LastTarget, c.Status.Generation)
}

Kubernetes cluster — HTTP API reference

The controller exposes a JSON REST API on port 80 (internal ClusterIP). All request and response bodies use Content-Type: application/json.

POST /timeshifts Create a timeshift 
// Request
{
  "namespace": "default",
  "labelSelector": "app=my-service",
  "time": "2030-01-01T00:00:00Z",
  "ttl": "1h",          // optional; omit for no expiry
  "freeze": false        // optional; true = pin clock at exactly "time"
}
// Response 201
{
  "id": "b3f5c2d1-...",
  "namespace": "default",
  "labelSelector": "app=my-service",
  "time": "2030-01-01T00:00:04Z",  // live effective time (advancing: grows each second)
  "frozen": false,
  "expiresAt": "2026-06-21T11:00:00Z",
  "appliedTo": ["pod/my-service-abc/app", "pod/my-service-def/app"]
}
GET /timeshifts List all active timeshifts 
// Response 200
{ "timeshifts": [ { "id": "...", "time": "...", ... } ] }
GET /timeshifts/{id} Get a timeshift by ID

Returns 404 if the timeshift does not exist or has expired. The time field reflects the live effective time — for advancing timeshifts it grows every second; for frozen timeshifts it is constant.

PATCH /timeshifts/{id} Update the target time or advance by a duration

Provide exactly one of time (absolute RFC3339) or duration (Go duration string). Both fields together return 400.

// Absolute update — move clock to a specific instant
{ "time": "2035-06-01T00:00:00Z", "freeze": false }

// Relative update — shift forward (or backward) by a duration
{ "duration": "24h" }      // advance by one day
{ "duration": "-1h30m" }   // rewind by 90 minutes

// Response 200 — updated timeshift object

The freeze field is optional and only meaningful with time. When using duration, the current mode (advancing or frozen) is preserved.

DELETE /timeshifts/{id} Delete a timeshift (resets all injected pods to real time)

Returns 204 No Content on success. Each injected container's clock is reset via process_vm_writev before the timeshift is removed from the registry.

GET /timeshifts/{id}/status Live injection status for each container 
// Response 200
{

  "namespace": "default",
  "containers": [

        "lastTarget": "2030-01-01T00:00:00Z",

      }

}

// Response 200
{
  "pods": [
    { "name": "my-service-abc", "namespace": "default", "nodeIP": "10.0.0.1",

GET /healthz Health check

{ "status": "ok" }
GET /metrics Prometheus metrics

Returns Prometheus-format metrics. The controller deployment includes annotations for Prometheus scraping. Key metrics:

  • epochd_timeshifts_active — gauge: current number of active timeshifts
  • epochd_inject_total — counter: successful injection calls
  • epochd_settime_total — counter: successful SetTime calls
  • epochd_sweep_expired_total — counter: timeshifts removed by TTL sweeper
  • epochd_api_requests_total{method,path,status} — counter: HTTP requests by route

Freeze mode

In freeze mode the target's clock is pinned to a specific instant. Every call to clock_gettime(CLOCK_REALTIME, ...) returns exactly that timestamp regardless of how much real time passes. The trampoline uses enabledMask = 3 (MaskFrozen) and stores the absolute frozen timestamp in the offset fields.

Freeze mode is useful for:

  • Reproducing time-sensitive bugs that only trigger at a specific instant.
  • Tests where the exact timestamp must match — e.g. certificate expiry checks where the serial number or expiry string is part of the assertion.
  • Pausing time during complex setup, then advancing by discrete steps.

Advancing mode

  • Clock runs at normal speed
  • Fake time = real time + fixed offset
  • Offset computed at injection time: target − now
  • GET response time field grows every second
  • Created with freeze: false (the default)

Frozen mode

  • Clock is completely stopped
  • Fake time = constant pinned instant
  • GET response time field is always the same value
  • Created with freeze: true
  • Can be unfrozen via PATCH with "freeze": false

Freeze via HTTP API

curl -X POST http://localhost:8080/timeshifts \
  -H 'Content-Type: application/json' \
  -d '{"namespace":"default","labelSelector":"app=web",
       "time":"2030-01-01T00:00:00Z","freeze":true}'

# Later — unfreeze and resume advancing from a new point:
curl -X PATCH http://localhost:8080/timeshifts/<id> \
  -H 'Content-Type: application/json' \
  -d '{"time":"2030-06-01T00:00:00Z","freeze":false}'

Freeze via CLI

faketimectl create --namespace=default --selector=app=web \
  --time=2030-01-01T00:00:00Z --freeze
# created timeshift a1b2c3d4-...
#   time:   2030-01-01T00:00:00Z
#   frozen: yes

# Unfreeze and move to a new time:
faketimectl update a1b2c3d4-... --time=2030-06-01T00:00:00Z

Freeze via pkg/faketime

frozen := time.Date(2030, 1, 1, 0, 0, 0, 0, time.UTC)

// Start a child process with its clock frozen.
h, err := faketime.StartFrozen(cmd, frozen)
if err != nil { t.Fatal(err) }
defer h.Reset()

// Advance the frozen clock by one day (still frozen at the new instant).
h.Advance(24 * time.Hour)

// Switch to advancing mode from a new base.
h.SetTime(time.Now().Add(365 * 24 * time.Hour))

Advance by duration

Instead of specifying an absolute target time, you can shift the current fake time by a relative amount. This works in both advancing and frozen modes and preserves the current mode.

💡

Use duration when you want to step through time in a test scenario without having to compute absolute timestamps at each step. For example: create a timeshift at now+1 year, then advance by 30 days, then advance by 6 months — each step is relative to where the clock currently is.

Advance via HTTP API

# Advance by 24 hours:
curl -X PATCH http://localhost:8080/timeshifts/<id> \
  -H 'Content-Type: application/json' \
  -d '{"duration":"24h"}'

# Rewind by 90 minutes:
curl -X PATCH http://localhost:8080/timeshifts/<id> \
  -H 'Content-Type: application/json' \
  -d '{"duration":"-1h30m"}'

Accepted Go duration strings: "30s", "5m", "24h", "72h", "-1h", "8760h" (one year). Providing both time and duration in the same request returns 400 Bad Request.

Advance via CLI

faketimectl advance <id> --by=24h      # forward one day
faketimectl advance <id> --by=-1h      # rewind one hour
faketimectl advance <id> --by=8760h    # forward one year

Advance via Go SDK

// Shift forward by one day.
ts, err := client.AdvanceTimeshift(ctx, ts.ID, 24*time.Hour)

// Rewind by 90 minutes.
ts, err = client.AdvanceTimeshift(ctx, ts.ID, -90*time.Minute)

Advance via pkg/faketime

// Single process:
h.Advance(24 * time.Hour)

// All processes in a session simultaneously:
s.Advance(24 * time.Hour)

Go tests (no cluster) — overview

pkg/faketime provides the same injection capability without any Kubernetes infrastructure. You import the package directly into your Go tests and inject into processes you start yourself — no agent, no controller, no cluster.

When to use

  • Testing a binary that calls time.Now(), reads clock_gettime, or uses time-based logic
  • Injecting multiple cooperating processes that should share a fake clock
  • CI environments where spinning up a k8s cluster is too heavy
  • Local development and rapid iteration

Constraints

  • Linux x86-64 only (same as the cluster path)
  • Requires either running the tests natively with CAP_SYS_PTRACE, or in Docker with --cap-add SYS_PTRACE --security-opt seccomp=unconfined
  • Only injects into processes you own — no injecting into arbitrary running PIDs without elevated permissions
💡

The stub in faketime_stub.go (compiled on non-Linux platforms) makes all functions return errors and all test helpers call t.Skip(). Your test code compiles and runs cleanly on macOS and Windows — the tests are simply skipped unless running on Linux.

Go tests (no cluster) — quickstart

Add the import

import "github.com/bkaznowski/epochd/pkg/faketime"

Install the standalone module:

go get github.com/bkaznowski/epochd/pkg/faketime

Inject into a process you start

Use faketime.Start to set a fake clock on a child process from the moment it begins executing:

func TestMyBinary(t *testing.T) {
    cmd := exec.Command("./my-binary", "--some-flag")
    // Do NOT call cmd.Start() — faketime.Start does it for you.

    target := time.Now().Add(365 * 24 * time.Hour) // one year in the future
    h, err := faketime.Start(cmd, target)
    if err != nil {
        t.Fatalf("faketime.Start: %v", err)
    }
    t.Cleanup(func() {
        h.Reset()             // snap back to real time
        cmd.Process.Kill()    // clean up the child
        cmd.Wait()
    })

    // The process is now running with its clock set to target.
    // Use cmd.Process.Pid, wait for output, etc.
}

Or use the WithProcess helper

WithProcess handles cleanup automatically via t.Cleanup, including on test failure:

func TestWithHelper(t *testing.T) {
    cmd := exec.Command("./my-binary")
    target := time.Now().Add(365 * 24 * time.Hour)

    faketime.WithProcess(t, cmd, target, func(t *testing.T, h *faketime.Handle) {
        // h.SetTime(newTarget) — update the fake time
        // h.Reset()            — snap back to real time
        // cmd is running; query it however you need
    })
    // Process is killed and time is reset before this line is reached,
    // even if the test calls t.Fatal.
}

Running the tests

On a Linux host with CAP_SYS_PTRACE:

go test ./... -count=1

In Docker (works on macOS and Windows):

docker run --rm \
  --cap-add SYS_PTRACE \
  --security-opt seccomp=unconfined \
  -v "$(pwd):/workspace" -w /workspace \
  golang:1.26-alpine \
  go test ./... -count=1
⚠️

Both flags are required in Docker. --cap-add SYS_PTRACE alone is insufficient — Docker's default seccomp profile blocks the ptrace syscall family. You must also pass --security-opt seccomp=unconfined.

Go tests (no cluster) — API reference

Handle — single process

Function / methodDescription
Start(cmd *exec.Cmd, target time.Time) (*Handle, error) Starts cmd in advancing mode. Sets cmd.SysProcAttr.Ptrace = true, calls cmd.Start(), injects via FollowChild (no elevated permissions required). Do not call cmd.Start() yourself.
StartFrozen(cmd *exec.Cmd, target time.Time) (*Handle, error) Starts cmd with its clock frozen at target. Every clock_gettime call in the child returns exactly target until SetTime or Freeze is called.
Attach(pid int, target time.Time) (*Handle, error) Injects into an already-running process in advancing mode using PTRACE_ATTACH. Requires CAP_SYS_PTRACE and ptrace_scope ≤ 1.
AttachFrozen(pid int, target time.Time) (*Handle, error) Injects into an already-running process with the clock frozen at target. Requires CAP_SYS_PTRACE and ptrace_scope ≤ 1.
h.SetTime(target time.Time) error Switches to advancing mode and sets the fake time. Uses process_vm_writev only (~1 µs, no ptrace stop).
h.Freeze(target time.Time) error Switches to frozen mode and pins the clock at target. Uses process_vm_writev only.
h.Advance(d time.Duration) error Shifts the current fake time by d (may be negative to rewind). Preserves the current mode — advancing handles stay advancing; frozen handles stay frozen at the new instant.
h.Reset() error Snaps the process back to the real clock. Equivalent to h.SetTime(time.Now()).

Session — multiple processes sharing one clock

A Session groups multiple processes under a single target time. SetTime issues all process_vm_writev calls concurrently to minimise the inter-process race window.

target := time.Now().Add(24 * time.Hour)
s := faketime.NewSession(target)

// Add processes via Start or Attach.
if err := s.Start(cmd1); err != nil { ... }
if err := s.Start(cmd2); err != nil { ... }

fmt.Println("processes in session:", s.Len()) // 2

// Move all processes to +48h simultaneously.
if err := s.SetTime(time.Now().Add(48 * time.Hour)); err != nil { ... }

// Reset all to real time.
if err := s.Reset(); err != nil { ... }
MethodDescription
NewSession(target time.Time) *SessionCreates an empty session in advancing mode with the given initial target.
s.Start(cmd *exec.Cmd) errorStarts the command and adds its handle to the session. Respects the session's current mode (advancing or frozen).
s.Attach(pid int) errorAttaches to a running process and adds it to the session. Respects the session's current mode.
s.SetTime(target time.Time) errorSwitches all handles to advancing mode and updates the time concurrently. Errors from individual handles are joined.
s.Freeze(target time.Time) errorSwitches all handles to frozen mode at target concurrently.
s.Advance(d time.Duration) errorShifts all handles by d concurrently, preserving the current mode.
s.Reset() errorResets all processes to real time.
s.Len() intReturns the number of handles in the session.

Test helpers

These functions are defined in the main package (not in _test.go), so they can be imported by other packages' test files. All use t.Cleanup instead of defer, so cleanup runs even when the passed function calls t.Fatal.

HelperDescription
WithProcess(t, cmd, target, fn) Starts cmd with fake time, calls fn(t, *Handle), then kills the process and resets the clock.
WithPID(t, pid, target, fn) Attaches to pid, calls fn(t, *Handle), then resets the clock. Requires CAP_SYS_PTRACE.
WithSession(t, target, setup, fn) Creates a session, calls setup(s) to populate it, calls fn(t, *Session), then resets and cleans up.

WithSession example

func TestTwoServicesShareClock(t *testing.T) {
    target := time.Now().Add(365 * 24 * time.Hour)

    faketime.WithSession(t, target,
        func(s *faketime.Session) error {
            if err := s.Start(exec.Command("./service-a")); err != nil {
                return err
            }
            return s.Start(exec.Command("./service-b"))
        },
        func(t *testing.T, s *faketime.Session) {
            // Both service-a and service-b see time ~1 year in the future.
            // s.SetTime(...) updates both simultaneously.
        },
    )
}

Fork & exec tracking — ChildTracker

When the target process spawns child processes via fork/vfork, the children inherit the patched vDSO mapping and see the same fake time automatically. However, if a child (or the parent itself) calls exec(), the new process image replaces the trampoline page and injection is lost.

StartWithTracking and StartFrozenWithTracking keep ptrace attached after the initial injection and run a background goroutine that handles both cases:

  • Fork events (PTRACE_EVENT_FORK / PTRACE_EVENT_VFORK) — a Handle is created for the child immediately, pointing at the same trampoline the child inherited. No re-injection needed.
  • Exec events (PTRACE_EVENT_EXEC) — the process is re-injected from scratch (new mmap, new trampoline, new vDSO patch) before it runs any user code. This covers both self-exec (same PID, e.g. PEX bootstrap) and fork+exec (child executes a new binary).
Function / methodDescription
StartWithTracking(cmd, target) (*ChildTracker, error) Starts cmd in advancing mode and returns a ChildTracker that automatically handles all descendant forks and execs. No elevated permissions required.
StartFrozenWithTracking(cmd, target) (*ChildTracker, error) Same as StartWithTracking but starts with the clock frozen at target.
ct.Handle The parent process's fake-time Handle. Call SetTime, Freeze, or Advance on it to update the parent's clock.
ct.Children() []*Handle Returns Handles for all child processes currently tracked. The list grows as new children are forked.
ct.Err() error Returns the first error the background loop encountered, if any. Check after Close.
ct.Close() error Stops the watch loop, resets all tracked children to the real clock, and detaches ptrace from the parent and all children. The parent process continues running.

Example — Postgres postmaster pattern

func TestPostgresTimeshift(t *testing.T) {
    cmd := exec.Command("postgres", "-D", "/tmp/pgdata")
    target := time.Now().Add(365 * 24 * time.Hour)

    ct, err := faketime.StartWithTracking(cmd, target)
    if err != nil {
        t.Fatalf("StartWithTracking: %v", err)
    }
    t.Cleanup(func() {
        ct.Close()        // detaches ptrace, resets children's clocks
        cmd.Process.Kill()
        cmd.Wait()
    })

    // ct.Handle controls the postmaster's clock.
    // Any backend processes forked by the postmaster are automatically tracked
    // and see the same fake time.
    // Advance all clocks by 30 days:
    if err := ct.Handle.Advance(30 * 24 * time.Hour); err != nil {
        t.Fatal(err)
    }
    // ct.Children() now includes handles for each forked backend.
}
💡

StartWithTracking keeps ptrace attached to the parent for the duration of tracking. This adds a brief ptrace-stop on each fork call (typically microseconds) while the child is registered. For long-lived child processes this overhead is negligible, but avoid it for processes that fork thousands of times per second.

CLI tool — faketimectl

faketimectl is a command-line tool for managing epochd timeshifts. It has two sets of subcommands: controller subcommands that talk to a running epochd controller over HTTP, and local injection subcommands that inject directly into a process via ptrace.

Build

GOOS=linux go build -o faketimectl ./cmd/faketimectl

General usage

faketimectl [--url=<controller-url>] <subcommand> [flags]

# The controller URL can also be set via environment variable:
export EPOCHD_URL=http://localhost:8080

These subcommands talk to a running epochd controller. Set EPOCHD_URL or pass --url to each command.

These subcommands inject directly into a process via ptrace. No epochd controller is required.

Security model

What epochd can do

epochd's mechanism — patching vDSO bytes in a running process via PTRACE_POKETEXT and writing to its memory via process_vm_writev — grants significant power:

  • The agent can inject arbitrary code pages into any process running on the node (because it has hostPID: true and CAP_SYS_PTRACE).
  • The controller can instruct any agent to target any container in the cluster (because the controller has cluster-wide pod list access).
  • There is no authentication on the controller's HTTP API in v1. Any pod that can reach the controller service can shift the clock of any other pod.

Security — concerns and mitigations

🔴

Deploy only in dedicated test clusters

The agent DaemonSet runs as a privileged pod with hostPID: true. An attacker who controls the controller API can instruct the agent to inject arbitrary code into any process on any node. Treat this as equivalent to having root on every node in the cluster.

No controller authentication (v1 limitation)

The controller's HTTP API has no authentication in the current version. Any pod that can reach the epochd-controller service (or any user with kubectl port-forward access) can create, update, and delete timeshifts for any pod in the cluster.

Mitigation options:

  • Network policy — restrict access to the controller service to only the namespaces and pods that run your e2e tests. This is the simplest and most effective control in the short term.
  • Dedicated namespace — deploy epochd in an isolated namespace with tight admission control.
  • Shared token (future) — the roadmap includes an optional --auth-token flag. Clients pass the token in an Authorization: Bearer header.

Agent privileges are broad

The privileged: true flag gives the agent essentially full access to the host. The recommended mitigation is to only deploy epochd in clusters that are already treated as low-trust dev/test environments, and to apply Kubernetes admission policies (OPA/Gatekeeper, Kyverno) that prevent privileged pods from being created in other contexts.

pkg/faketime — local privilege requirements

When using pkg/faketime in Go tests:

  • faketime.Start uses the FollowChild path (PTRACE_TRACEME from the child). This does not require elevated privileges — you don't need CAP_SYS_PTRACE or root. It works in Docker with only --cap-add SYS_PTRACE --security-opt seccomp=unconfined.
  • faketime.Attach uses PTRACE_ATTACH and does require CAP_SYS_PTRACE and ptrace_scope ≤ 1. Prefer Start when you can control the target process's launch.

A future release will optionally intercept CLOCK_MONOTONIC.
No clock uninstall The vDSO patch and trampoline page remain for the life of the process. There is no Uninstall() that restores original bytes. h.Reset() / DeleteTimeshift snap the clock back to real time using a single process_vm_writev call. For almost all test scenarios this is equivalent to a full uninstall.
Re-injection leaks a trampoline page Calling InjectAtTime on the same PID twice allocates a second 4 KB page (the old one is not unmapped). Acceptable for testing. The agent re-injects on handle recovery after a restart and does not do so repeatedly in normal operation.
Race on time reads The trampoline reads the state struct with plain word loads. A SetTime call racing with a clock_gettime call can return a time between the old and new offsets. The worst case is one slightly wrong timestamp. For test assertions, either read a timestamp after a short pause post-SetTime, or make the fake offset large enough that any in-between value is still clearly in the expected range.
Forked children inherit the injection Processes forked after injection share the same vDSO mapping and see the same fake time. This is usually desirable but the caller has no handle to update the child's clock independently. Use faketime.StartWithTracking to obtain per-child Handles and automatically re-inject after any exec() call. For simple cases where the same fake time is sufficient, the inherited injection is enough.
No HA for the controller The timeshift registry is in-memory and per-process. A controller pod restart causes all active timeshifts to reload from the ConfigMap backup, but handle state in agents is lost and must be re-injected. Controller restart recovery (Phase 20) persists the registry to a ConfigMap. After a restart the controller re-injects via its pod watcher. Brief gaps (< one sweep interval, default 30s) are possible during a restart.
Single containerd runtime Container ID → PID resolution reads /proc/*/cgroup and matches the container ID string. This works for containerd but has not been tested with CRI-O or Docker's CRI shim. The resolution path is in pkg/k8sresolve and can be extended for other runtimes.

FAQ

Does this work with Go's time.Now()?

Yes. Go's time.Now() calls clock_gettime(CLOCK_REALTIME, ...) via the vDSO. epochd patches exactly that entry point, so time.Now() returns the fake time after injection.

Will time.Since / time.Until work correctly?

Yes and no. time.Since(t) and time.Until(t) in Go are computed from the monotonic component of the time (which is stored when time.Now() is called). If both the reference time and the current time are read after injection, the difference will be correct. If one was read before injection and the other after, the result may be surprising. For most test patterns (read a reference time, advance the clock, check relative to the reference) this works correctly.

CLOCK_MONOTONIC itself is not intercepted, so timeouts and rate limiters that use the monotonic clock internally won't be affected by the injection.

Can I use this with Python, Node.js, or Java?

For Kubernetes-deployed services: yes. The controller API is plain HTTP+JSON. Any language can use POST /timeshifts to shift the clock of any container regardless of what language that container is written in.

For local process injection: only via the Go pkg/faketime API or the faketimectl CLI.

What happens if the target process calls fork?

Forked children inherit the parent's vDSO mapping, which includes the patched JMP and the trampoline page. Children see the same fake time as the parent. This is usually the desired behaviour when testing a multi-process server.

If you need a Handle for each child (to update clocks independently, or to track when children exit), use faketime.StartWithTracking instead of faketime.Start. It runs a background goroutine that watches for PTRACE_EVENT_FORK notifications and registers a Handle for each child as it is created.

What happens if the target process calls exec?

exec() replaces the entire process image. The kernel remaps the vDSO clean and the trampoline page is gone — the process returns to seeing real time.

Use faketime.StartWithTracking to handle this automatically. The background goroutine listens for PTRACE_EVENT_EXEC stops (fired before the new binary's _start runs) and performs a fresh injection into the new address space. This covers both self-exec (same PID, e.g. a PEX bootstrap calling os.Execv on itself) and fork+exec patterns (a child process executing a new binary).

What if the agent restarts?

The agent holds the inject handle map in memory. When it restarts, the handles are gone. The controller detects this: a failed gRPC call returns "handle not found", which triggers re-injection. Additionally, the pod watcher detects the agent's container ID changing and proactively re-injects all affected timeshifts. A brief window (typically under a few seconds) where the target process sees real time is possible during an agent restart.

Can two timeshifts target the same pod?

No — the controller prevents it. If any container matched by a POST /timeshifts request is already owned by an active timeshift, the request is rejected with 409 Conflict and a message identifying the conflicting containers and the timeshift ID that owns them. Delete or update the existing timeshift first.

In the SDK: sdk.IsConflict(err) reports whether an error is a 409 response, so you can handle it explicitly:

_, err := client.CreateTimeshift(ctx, ns, selector, target, ttl)
if sdk.IsConflict(err) {
    log.Printf("conflict: %v", err)
    // list existing timeshifts to find the owner, or delete it first
}

What's the difference between freeze mode and advancing mode?

In advancing mode (the default), the trampoline adds a fixed offset to the real time returned by the syscall. The clock ticks forward at the normal rate — if you inject +2 years and check time.Now() one second later, you'll see a time that is 2 years and 1 second ahead.

In freeze mode, the trampoline ignores the real syscall result entirely and returns a constant absolute timestamp. No matter how many seconds of real time pass, time.Now() in the target always returns exactly the frozen instant. Use freeze mode when the exact timestamp matters — for example, when testing a certificate whose serial number or expiry must match a specific value, or when you need full control over each time step in a test scenario.

Can I advance a frozen clock step by step?

Yes. Use AdvanceTimeshift (SDK), faketimectl advance --by=... (CLI), or h.Advance(d) (pkg/faketime). Each advance shifts the pinned instant by the given duration and the clock remains frozen at the new value. This is useful for simulating discrete time steps (e.g. "now advance by one billing cycle") without switching to advancing mode.

Is there a Helm chart?

Not yet. The deploy/ directory contains raw Kubernetes manifests. A Helm chart is on the roadmap.

Will this work on kind, minikube, or k3s?

Yes, with the right configuration. The cluster must allow privileged pods. On kind, the default configuration allows this. On minikube, you may need to pass --extra-config=apiserver.enable-admission-plugins=PodSecurityAdmission and configure the policy to allow privileged pods in the epochd namespace. The e2e tests in the repo target a kind cluster via the Makefile.
